# Pixel Lab Pro 项目中期技术调研报告

> 文档用途：用于课程项目中期汇报，说明项目当前技术方案、关键实现、风险评估与下阶段计划。

## 1. 项目背景与中期目标

### 1.1 项目定位
Pixel Lab Pro 是一个“数字图像创作与管理平台”，面向课程作业场景，覆盖用户认证、图像处理、社区互动与后台管理四类核心能力。

### 1.2 中期里程碑达成情况
截至中期，项目已完成以下主线能力：
- 前后端工程搭建与基础规范统一（Vue3 + Vite + Express）。
- 账号体系闭环（注册/登录/鉴权/资料与密码管理）。
- 图像上传与管理能力（上传、列表、删除、可见性控制）。
- 社区域模型落地（点赞、收藏、评论、动态）。
- 管理端核心能力（用户管理、作品管理、平台统计）。

---

## 2. 技术架构调研与选型结论

## 2.1 总体架构
项目采用经典前后端分离架构：
- 前端：Vue 3 SPA，负责交互、状态管理与可视化展示。
- 后端：Express REST API，负责业务编排、鉴权、文件处理与数据访问。
- 数据层：MySQL，使用结构化表支撑账号与社区关系数据。

该架构在课程项目场景中的优势：
1. 上手快，团队分工明确（前后端并行）。
2. 技术生态成熟，排障资料丰富。
3. 易于中后期扩展为“多端 + 统一 API”。

### 2.2 前端技术调研结论
当前前端技术栈：Vue 3、Vite、Element Plus、Vue Router、Pinia、Axios、ECharts、Fabric。

**选型理由与收益：**
- **Vue 3 + Composition API**：组件逻辑复用与组织能力更强，适合工作台复杂交互。
- **Vite**：开发热更新速度快，工程启动和构建体验更好。
- **Element Plus**：大幅减少后台与表单页面的 UI 开发成本。
- **Pinia + Router 守卫**：统一状态与权限入口，降低页面级重复判断。
- **Axios 二次封装**：统一 token 注入和异常处理，减少接口调用样板代码。

### 2.3 后端技术调研结论
当前后端技术栈：Express、mysql2、JWT、bcryptjs、Joi、multer、sharp。

**选型理由与收益：**
- **Express**：轻量、灵活，适合课程项目快速交付。
- **mysql2**：性能与稳定性较好，支持连接池。
- **JWT**：适合前后端分离鉴权，服务端无状态扩展方便。
- **Joi + 中间件**：对输入进行前置校验，降低脏数据进入业务层概率。
- **multer + sharp**：覆盖上传与图像处理基础能力，为后续压缩/缩略图扩展预留空间。

### 2.4 数据库方案调研结论
数据库采用 MySQL，字符集为 utf8mb4，适配中文与 Emoji 文本。社区能力采用“主表 + 关系表”的结构设计（作品、点赞、评论、收藏），满足中期查询与统计需求。

**中期评价：**
- 关系模型清晰，支持后续分页、排行、热门推荐等常见查询。
- 社区统计字段（like/comment/collect/view）可减少高频列表页聚合开销。

## 3. 关键模块实现分析（中期现状）

### 3.1 认证与权限控制
- 路由层已区分公开接口和鉴权接口。
- 前端通过路由守卫控制登录态与管理员权限页面。
- 后端通过 `authMiddleware`、`adminMiddleware` 执行接口级权限控制。

**价值：** 已形成“前端拦截 + 后端兜底”的双层权限机制。

### 3.2 图像业务链路
- 具备上传接口，接入鉴权和上传校验中间件。
- 支持用户维度图片列表、删除、可见性更新。
- 为社区展示提供资源基础（公开作品入口）。

**价值：** 核心“创作资产”链路已经跑通，是后续编辑器增强的基础。

### 3.3 社区互动能力
- 已实现公开浏览、详情、评论、点赞、收藏等关键接口。
- 已支持可选鉴权（未登录可浏览，登录可看到互动状态）。

**价值：** 社区域已形成可演示闭环，具备产品化雏形。

### 3.4 管理后台能力
- 后端已完成管理员路由统一鉴权。
- 支持用户状态/角色管理、作品管理与统计查询。

**价值：** 已具备基础运营与内容治理能力，满足课程答辩中的“平台治理”要求。

## 4. 工程质量与非功能性指标评估

### 4.1 已具备的工程保障
- 分层结构清晰（router/controller/service/middleware/utils）。
- 全局错误处理中间件已接入。
- Axios 请求与响应拦截统一处理。
- 基础 SQL 初始化脚本可复现数据库。

### 4.2 中期发现的问题
1. **安全默认值风险**：JWT 存在默认 secret 回退值，生产环境需强制校验。
2. **配置治理不足**：环境变量样例与实际配置项约束仍可加强（如必填校验）。
3. **可观测性较弱**：缺少结构化日志、接口耗时统计、链路追踪能力。
4. **自动化测试不足**：当前以功能联调为主，缺少单元测试/接口测试基线。

## 5. 性能与扩展性调研结论

### 5.1 当前瓶颈预判
- 图片上传与处理为高 I/O 场景，若并发上升会受磁盘与 CPU 影响。
- 社区列表页在数据增长后，排序与过滤会依赖更精细索引设计。
- 前端工作台若叠加更多 Fabric 操作，低性能设备可能出现卡顿。

### 5.2 可行优化方向（按优先级）
**P1（短期）**
- 为高频查询字段补全索引并复盘慢 SQL。
- 上传链路增加文件体积与尺寸前置限制。
- 接口增加分页上限和防刷策略。

**P2（中期）**
- 静态资源接入对象存储 + CDN（解耦应用服务与文件服务）。
- 引入 Redis 做热门数据缓存与登录态辅助能力。
- 前端编辑器引入 Web Worker 处理部分耗时计算。

**P3（长期）**
- 后端模块化拆分（认证、社区、资源服务）。
- 建立消息队列处理异步任务（缩略图、审核、统计汇总）。

## 6. 技术风险清单与应对策略

| 风险项 | 影响 | 当前状态 | 应对方案 |
|---|---|---|---|
| JWT 密钥配置不规范 | 高 | 待加固 | 启动时校验必填，不允许默认值上线 |
| 图片存储仅本地磁盘 | 中 | 可用 | 预研 OSS/S3 兼容方案，抽象存储接口 |
| 测试覆盖率不足 | 高 | 薄弱 | 建立 smoke + 核心 API 自动化用例 |
| 社区数据膨胀后查询变慢 | 中 | 潜在 | 索引优化 + 缓存层 + 归档策略 |
| 管理操作审计缺失 | 中 | 待补齐 | 增加管理员操作日志与追溯能力 |
